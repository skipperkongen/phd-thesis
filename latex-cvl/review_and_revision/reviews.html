List of Strong Points The paper provides a solution of map generalization, i.e. which objects of a set of objects should be displayed at a certain zoom level.
It introduces a descriptive language (CVL) allowing the user to define constraints and to create a subset of objects to display at a certain zoom level. These constraints are used for automatic filtering the source object set.
The problem mentioned above is mapped to an optimization problem. Two heuristics are presented for creating good results. Both, CVL and the heuristics are implemented on top of a relational database system with spatial extension. Experiments show that both heuristics provide reasonable results for real and synthetic data sets.
List of Weak Points The runtime of the linear programming based heuristic is very high for some constraints.
A central example (including the relation scheme) is missing.
Detailed Comments When displaying a set of geographic objects in a map at a certain zoom level, two problems
have to be solved, which objects should be displayed and how these objects should be displayed.
The paper describes a method to solve the first problem. Objects are stored in a relational
database with a spatial extension.
The goal is to assign each object a range of zoom levels in which the object should be part of the map.
For a large set of objects it's unfeasible to do this manually. The authors present a descriptive
language (CVL) allowing to define of constraints and to create a subset of objects to display at
a certain zoom level. Some extensions of this language are presented to make the use of CVL easier.
For filtering out objects from the source set, a weight is assigned to each object describing its importance.
Then, automatically finding a filter is mapped to an optimization problem. The authors state
that this optimization problem is too hard to solve and present two heuristics. Both of them
transfer the complete optimization problem into a series of smaller optimization problems (one for each zoom level)
and are mostly computable in reasonable time.
The "static greedy algorithm" (SGA) removes the objects having the smallest weight from the conflicting sets.
It is relatively fast and provides good results at least for point data.
The second algorithm (LPGA) uses linear programming to reach better results than SGA but takes more time.
Both algorithms and a CVL compiler are implemented on top of a PostGIS database system.
Several datasets are used to evaluate the approach. The experiments show that the runtime strongly depends
on the data type (point or other geometries) and the used constraints (visibility, proximity).

Automatic selection of objects for a map representation is an important issue. The paper is well written and
comprehensive. The experiments show, that the proposed technique creates good results but the runtime is
too high in some cases. I suggest to accept the paper.

List of Strong Points This paper has a great central idea: to express map rendering constraints in a declarative way, and to use a SQL database to execute those constraints. This is an underexamined area and this is, at a high-level, exactly the right approach.

It's also right to map this problem to some form of optimization task.

Finally, the set of experimental datasets and tasks seems right. I liked the expts about scalability and varying shapes.
List of Weak Points This paper makes all the right decisions at a high level, but has a number of problems at a lower level.

-- The decision to use conflict sets is underexamined. How many conflict sets can there be? How large can each one be? Can they characterize every possible desirable violation? These are all critical questions that the paper does not examine. (At one point the paper says that a conflict set can be quite large, but there is no complexity analysis, and the other questions do not appear to be examined)

-- Is a tile-based visibility constraint sufficient? Isn't it possible to have two adjacent tiles that independently do not violate a desired constraint, but which together do violate it? (Eg, if the left side of one tile is filled with objects, and the right side of a neighboring tile is also filled.) There is no argument that tiles are sufficient to capture this goal

-- The mapping of conflict sets to set cover is not done convincingly. This mapping should be much more carefully written, with examples provided.

-- Even if the mapping is possible, it's not clear that it's *necessary*. Isn't there possibly a better way to solve the map-specific conflicts? No discussion is here right now.
Detailed Comments Unclear: "Unfortunately, with current tools, ..." The notion of a zoom level isn't obvious yet.

Reword "symbolization"

Section II: it seems you do not handle "regional labels" that might reappear at more general zoom levels. (Eg, "Europe" reappears when the camera pulls out far enough.) That should be made explicit in the text.

As mentioned above, conflict sets have to be much more thoroughly examined. They are core to the paper but are treated quite casually.

Not sure what you mean here: "while also specifying how records are rendered"

"The create constraint... and the generalize statement is used for everything else." What is 'everything else'?

Section 3A: the notion of "weight" is casually introduced. Is it good or bad to have heavy weight? I eventually figured it out, but be clear here.

Not sure why the "AT X ZOOM LEVELS" clause is useful. Can you provide a motivating example? Current examples don't make the case. E.g., why "18" in Figure 4?

The paragraph, "The resulting map is shown in Figure 3..." this is exactly the opposite of what I thought would happen. If the zoom level is very general, won't it be made up of many tiles that are fixed in their number of pixels? Wouldn't each such tile have 16 airports, instead of 16 for the whole world?

The syntax for CVL isn't pretty. Even a simple example in Fig 6 takes a lot of clauses. Is this really the best way to do it? Figure 7 is worse, with a strange 'busted table' involved. Why is busted table needed in this example, but not Fig 6? Is it because it's zoom-level-independent?

The mapping of context sets to set cover is really critical and is not clear. Why can a conflict set be chosen only once?

Do the "extensions" add anything to the difficulty or time complexity? There's no discussion of whether this could happen.

List of Strong Points 1) a very detailed, clear and elegant presentation. All the technical details are explained in a very comprehensible text.
2) The proposed constraint solver is sound and well motivated
3) the language proposed to drive the solver seems easy to be adopted even for new users
List of Weak Points 1) Some generalization solutions use a change of geometry when changing zoom level. For example, roads are represented as lines at low zoom levels, and with polygons on high zoom levels. The authors may discuss how this aspect may be included as a new feature in their system.
2) it is not clear if the framework is available for the community as DBMS plugin. From the text it specified that an instance of the system is available as a virtual machine image.
Detailed Comments The authors present a framework to generalize geometries for large maps visualization. The proposed framework present a language as an extension of SQL to let the analyst to specify different families of contraints. The statements are interpreted by a compiler and passed to a constraint solver that removes conflicting entities.
The problem seems very interesting and the proposed solution is very elegant. The formal presentation of the problem is clear and well documented.

I'm not an expert of the field and thus I can not evaluate the novelty of the approach.
